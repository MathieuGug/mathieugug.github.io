# Outline — Désagrégation prefill/decode

## Thèse centrale
La désagrégation prefill/decode fait passer le serving LLM d'un problème de *scheduling* sur GPU homogènes à un problème d'*architecture distribuée* : deux pools de calcul aux profils matériels opposés (compute-bound vs memory-bound), reliés par un transfert de KV cache qui est à la fois le déverrouillage (goodput ×7) et le nouveau goulot d'étranglement.

## Plan (7 sections)
1. **L'interférence prefill/decode** — deux phases, deux profils ; la colocalisation classique (continuous batching) les fait se gêner mutuellement : un long prefill bloque les decodes (pic de TTFT), les decodes diluent l'utilisation compute du prefill. [SCHEMA-01]
2. **Goodput, pas throughput** — la métrique qui change tout. TTFT vs TPOT, deux SLO orthogonaux. Le throughput brut ment ; seules comptent les requêtes qui respectent les deux SLO. [SCHEMA-02]
3. **L'anatomie de la désagrégation** — deux pools indépendants, un routeur KV-aware, le transfert de KV cache, autoscaling séparé. [SCHEMA-03]
4. **Les trois écoles** — DistServe (goodput académique, OSDI 2024), Splitwise (efficacité énergétique/coût, Microsoft ISCA 2024), Mooncake (KVCache-centric, production Kimi/Moonshot). [SCHEMA-04]
5. **Le KV cache comme nouveau goulot** — NIXL, la hiérarchie de transport (NVLink intra-nœud, RDMA/IB inter-nœud, NVMe, object store), transfert non-bloquant layer-by-layer. [SCHEMA-05]
6. **Le contre-argument** — chunked prefill / Sarathi-Serve : la voie « agrégée » qui rivalise sans désagréger. Quand désagréger vaut-il la peine ? Désagrégation conditionnelle.
7. **La bascule production 2026** — llm-d (CNCF Sandbox mars 2026), Dynamo, vLLM production-stack. Standardisation + horizon 2026-2028. [SCHEMA-06]
