# Outline — Désagrégation prefill/decode

## Thèse centrale
L'inférence LLM fait coexister deux régimes de calcul antagonistes — le *prefill* (lecture du prompt, compute-bound) et le *decode* (génération token par token, memory-bound). Les co-loger sur la même machine fait que l'un sabote la latence de l'autre. La désagrégation — séparer prefill et decode sur des pools matériels distincts et transporter le KV cache sur le réseau — est le tournant architectural de l'inférence 2024-2026, de Splitwise (ISCA) et DistServe (OSDI) aux frameworks de production (Mooncake/Kimi, NVIDIA Dynamo, llm-d). Son rival n'est pas le statu quo mais le *chunked prefill* co-localisé (Sarathi-Serve) : le débat structurant de la couche serving.

## Plan (8 sections)

1. **Synthèse exécutive** — 5 bullets : deux régimes antagonistes ; l'interférence prefill↔decode ; la désagrégation comme réponse + son rival chunked prefill ; le KV cache devient objet réseau de première classe ; goodput sous SLO comme métrique ; production 2026.
2. **Deux régimes, une machine** — roofline. Prefill = un seul forward pass sur tout le prompt, parallélisable, compute-bound (sature les FLOPS). Decode = un token à la fois, autoregressif, relit tout le KV cache à chaque pas, memory-bandwidth-bound. Arithmetic intensity opposée. [SCHEMA-01 roofline]
3. **L'interférence : le péché du batch mixte** — continuous batching (Orca) mélange prefill et decode dans le même batch → un long prefill bloque les decodes en cours (generation stall / TBT spikes). Deux écoles de réponse : (a) co-localisé mais discipliné = chunked prefill (Sarathi-Serve, stall-free batching) ; (b) désagrégé. [SCHEMA-02 interférence + deux écoles]
4. **L'architecture désagrégée canonique** — pool prefill → KV cache transfer → pool decode ; routeur/scheduler global ; flux d'une requête. Splitwise + DistServe comme fondations. [SCHEMA-03 architecture]
5. **Le KV cache comme objet réseau** — taille (GB par requête), transport (RDMA/InfiniBand/NVLink, NIXL, Mooncake Transfer Engine), streaming layer-by-layer pour recouvrir transfert et calcul, pool tiéré CPU/DRAM/SSD (Mooncake), prefix caching/réutilisation. [SCHEMA-04 KV cache transfer + tiering]
6. **Goodput sous SLO : pourquoi désagréger paie** — TTFT (gouverné par prefill) vs TPOT/ITL (gouverné par decode) ; SLO distincts ; désagrégation permet de scaler prefill et decode *indépendamment* (ratio xP:yD), choisir parallélisme et matériel par phase. Métrique goodput (DistServe). Quand chunked prefill gagne (petits déploiements, charge homogène). [SCHEMA-05 goodput / scaling indépendant]
7. **Cartographie des systèmes 2026** — matrice : Splitwise, DistServe, TetriInfer, Mooncake, Sarathi-Serve(colocated), NVIDIA Dynamo, llm-d, vLLM(disagg expérimental), SGLang × dimensions (approche, transport KV, scheduling, statut prod, matériel). [SCHEMA-06 matrice]
8. **Trajectoires 2026-2028 + conclusion** — matériel hétérogène par phase (prefill GPU compute-dense, decode mémoire-dense/moins cher), KV cache as a service partagé multi-tenant, désagrégation × decode spéculatif, désagrégation des experts MoE, standardisation du transport (NIXL). Conclusion : la désagrégation déplace le goulot du GPU vers le réseau ; gagne à l'échelle, pas en deçà.

## Sources (12, tier-1)

2. Zhong et al., « DistServe: Disaggregating Prefill and Decoding for Goodput-optimized LLM Serving », OSDI 2024 / arXiv:2401.09670 (PKU + UCSD) — formalise le goodput sous SLO TTFT+TPOT, scaling indépendant, placement.
3. Qin et al., « Mooncake: A KVCache-centric Disaggregated Architecture for LLM Serving », FAST 2025 best paper / arXiv:2407.00079 (Moonshot AI / Kimi) — archi production, pool KV tiéré CPU/DRAM/SSD, Transfer Engine, overload-oriented scheduling.
4. Agrawal et al., « Taming Throughput-Latency Tradeoff in LLM Inference with Sarathi-Serve », OSDI 2024 / arXiv:2403.02310 (Microsoft Research India / GaTech) — le rival co-localisé : chunked prefills + stall-free batching.
5. Yu et al., « Orca: A Distributed Serving System for Transformer-Based Generative Models », OSDI 2022 — continuous/iteration-level batching, origine du batch mixte et de l'interférence.
6. Kwon et al., « Efficient Memory Management for LLM Serving with PagedAttention » (vLLM), SOSP 2023 / arXiv:2309.06180 — gestion mémoire KV, socle des serveurs modernes.

8. Strati et al., « DéjàVu: KV-cache Streaming for Fast, Fault-tolerant Generative LLM Serving », ICML 2024 / arXiv:2403.01876 — streaming KV, micro-batch swapping, tolérance aux pannes.
9. NVIDIA Dynamo — annonce GTC mars 2025, framework de serving open-source avec disaggregated serving + KV-aware routing + NIXL (docs NVIDIA / GitHub ai-dynamo).
10. llm-d — Red Hat / Google / IBM / CoreWeave, mai 2025, inférence distribuée Kubernetes-native (vLLM), disaggregation + KV-cache-aware routing (llm-d.ai).
11. Patel et al. / Microsoft, « Splitwise » blog Azure + POLCA power — angle coût/énergie/matériel hétérogène par phase.
12. vLLM & SGLang docs — disaggregated prefilling (KV connector, 1P1D) ; SGLang PD disaggregation + intégration Mooncake Transfer Engine — l'adoption open-source.

## Schémas (6) — avec régions cliquables
- **SCHEMA-01 — Le roofline des deux régimes.** Plot roofline (intensité arithmétique × débit), prefill sur le plateau compute, decode collé au mur mémoire. Régions : (a) zone prefill compute-bound, (b) zone decode memory-bound, (c) point d'inflexion (ridge), (d) plafond matériel (HBM bandwidth).
- **SCHEMA-02 — L'interférence et les deux écoles.** Timeline d'un batch mixte avec un prefill qui stalle les decodes ; puis deux branches : chunked prefill vs désagrégation. Régions : (a) le stall, (b) chunked prefill, (c) désagrégation, (d) métrique TBT/ITL.
- **SCHEMA-03 — Architecture désagrégée canonique.** Routeur → pool prefill → transfert KV → pool decode → flux de tokens. Régions : (a) routeur/scheduler global, (b) pool prefill, (c) transfert KV, (d) pool decode.
- **SCHEMA-04 — Le KV cache comme objet réseau.** Anatomie du KV cache + transports + pool tiéré + streaming layerwise. Régions : (a) taille/anatomie KV, (b) transports RDMA/NVLink/NIXL, (c) pool tiéré CPU/DRAM/SSD (Mooncake), (d) streaming layer-by-layer.
- **SCHEMA-05 — Goodput sous SLO.** Deux axes TTFT/TPOT, frontière goodput, levier de scaling indépendant xP:yD. Régions : (a) axe TTFT/prefill, (b) axe TPOT/decode, (c) levier scaling indépendant, (d) zone où chunked prefill gagne.
- **SCHEMA-06 — Cartographie des systèmes.** Matrice 9 systèmes × 5 dimensions. Régions : (a) école désagrégée pure, (b) école co-localisée (Sarathi), (c) frameworks production (Mooncake/Dynamo/llm-d), (d) adoption open-source (vLLM/SGLang).
