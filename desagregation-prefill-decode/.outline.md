# Outline — Désagrégation prefill / decode

## Thèse centrale
Le *prefill* (compute-bound, parallèle sur tout le prompt) et le *decode* (memory-bound, séquentiel token par token) sont deux charges aux profils matériels antagonistes que l'histoire a forcées à partager la même puce. Les séparer en pools distincts reliés par une fabrique de transfert de KV cache — la désagrégation prefill/decode — re-optimise chaque phase indépendamment et débloque 2 à 7× de *goodput* sous SLO. Elle fait du KV cache, et non plus du GPU, le centre de gravité du système d'inférence.

## Plan (8 sections)
1. **Synthèse exécutive** — bullets chiffrés (interférence, goodput, KV centre de gravité, continuum chunked-prefill, état 2026).
2. **Le péché originel : deux charges antagonistes sur une même puce** — roofline ; prefill compute-bound (haute intensité arithmétique, GEMM sur tout le prompt) vs decode memory-bound (un token, borné par la bande passante HBM lecture poids + KV) ; l'interférence (un long prefill bloque les decodes → pic de TPOT) ; le double couplage : parallélisme (TP optimal prefill ≠ decode) et matériel (decode gaspille les FLOPS d'un H100). [SCHEMA-01]
3. **L'anatomie de la désagrégation** — pools séparés, routeur/scheduler, transfert KV. DistServe (7,4× requêtes / 12,6× SLO plus serré) et Splitwise (2,35× throughput à coût/puissance égaux). [SCHEMA-02]
4. **Le KV cache devient le centre de gravité** — Mooncake KVCache-centric, pool DRAM/SSD des ressources sous-utilisées, NIXL/RDMA VRAM↔VRAM, transfert pipeliné couche par couche (overlap compute/I/O), overhead réel (TTFT ×1,4-1,9 sur petits prompts), réutilisation de préfixe. [SCHEMA-03]
5. **Goodput, pas throughput : la métrique qui change tout** — SLO TTFT/TPOT, goodput = requêtes servies dans la boîte SLO ; rééquilibrage dynamique du ratio P:D selon la charge ; pourquoi le throughput agrégé ment. [SCHEMA-04]
6. **Le rival intra-instance : chunked prefill (Sarathi-Serve)** — découpe du prefill en chunks interleavés avec les decodes (stall-free, 5,6×) ; le continuum désagréger ⇄ multiplexer ; point de bascule (échelle du déploiement, longueur de prompt, hétérogénéité du matériel). [SCHEMA-05]
7. **L'état de l'art 2026 : qui désagrège quoi** — matrice DistServe / Splitwise / Mooncake / Dynamo / vLLM (llm-d) / SGLang / TensorRT-LLM ; matériel hétérogène (prefill H100/B200 FLOPS, decode H200 141 Go HBM3e) ; déploiements prod (Kimi K2 128×H200 : 224k prefill / 288k decode tok/s ; DeepSeek 96×H100 52,3k in / 22,3k out par nœud). [SCHEMA-06]
8. **Trajectoires 2026-2028** — désagrégation à grain plus fin (attention disaggregation / Adrenaline, PPD multi-tour) ; cross-vendor (H100 + MI300X) ; KVCache global / cache-as-a-service ; couplage désagrégation × MoE expert-parallel. [SCHEMA-07]
9. **Conclusion** (court).
