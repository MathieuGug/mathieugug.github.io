Évaluer un agent

Le playbook construit les défenses · le gruyère arrête les attaques

On construit l'évaluation en 4 phases (collecte tasks, cadrage, instrumentation, maintenance) — c'est le playbook. Le résultat est un dispositif à 3 couches de défense (préventif, curatif, qualitatif) — c'est le gruyère de Reason. Chaque couche a ses trous, seul l'empilement arrête les attaques.

« Chaque couche a ses trous ; l'empilement seul produit une couverture acceptable. » — James Reason, modèle gruyère suisse

Le playbook — comment on construit les défenses

4 phases dans le temps · chaque phase produit ou maintient une couche du dispositif.

Collecte tasks

S0

Collecte tasks · démarrer tôt

Démarrer tôt

20–50 tasks de vrais échecs suffisent à amorcer un système d'eval

  • µ-1
    20–50 tasks

    volume initial

    assez pour révéler 80 % des modes d'échec

  • µ-2
    Issues de vrais échecs

    sources

    bugs reportés, edge cases déjà observés en prod

  • µ-3
    Règle 80/20

    discipline

    on capture l'essentiel avec un effort modeste

  • µ-4
    Value compose

    effet cumulatif

    chaque task ajoutée renforce le harness existant

⚠ ANTI-PATTERN

Attendre 1 000 tasks gold-labellisées avant de démarrer

Conséquence : la dette d'eval gonfle pendant qu'on optimise dans le noir.

Phase 2A · cadrer les cas

Cadrer les cas

S2

Cadrer les cas · non-ambiguïté

Non-ambiguïté

Tasks dont 2 experts conviennent sur pass/fail · pass@k vs pass^k

  • µ-1
    2 experts → même verdict

    critère minimal

    si pas d'accord, la task est cassée

  • µ-2
    Reference solution

    gold standard

    la "bonne" réponse documentée

  • µ-3
    pass@k

    au moins 1 succès sur k tries

    mesure de la capabilité brute

  • µ-4
    pass^k

    tous les k tries réussis

    mesure de fiabilité (plus stricte)

⚠ ANTI-PATTERN

0% pass = task cassée

Si même les meilleurs experts ne savent pas trancher, le grader ne saura pas.

S3

Cadrer les cas · équilibre

Équilibre

Should trigger / should NOT trigger — éviter l'overtriggering

  • µ-1
    should trigger

    cas où l'agent doit agir

    mesure la capability

  • µ-2
    should NOT trigger

    cas où l'agent doit ne PAS agir

    mesure la prudence

  • µ-3
    Ratio cible

    50/50

    pour éviter le bias vers une réponse par défaut

  • µ-4
    Diversité

    angles d'attaque variés

    pas tous les cas du même domaine

⚠ ANTI-PATTERN

Web search : 90% should-trigger

L'agent apprend à toujours déclencher → coût, latence, et faux positifs.

S5

Harness & graders · graders sobres

Graders sobres

Déterministe > LLM-as-judge > humain · grader le résultat pas le chemin

  • µ-1
    Hiérarchie 1

    déterministe quand possible

    regex, schema check, exact match

  • µ-2
    Hiérarchie 2

    LLM-as-judge pour le nuancé

    avec calibration humaine

  • µ-3
    Hiérarchie 3

    humain pour les cas durs

    gold standard, mais lent et cher

  • µ-4

    éviter le tout-ou-rien

    gradient de qualité = signal plus riche

⚠ ANTI-PATTERN

Si le LAJ n'est pas calibré, le bruit masque le signal. Perte de confiance.

Lire transcripts

Saturation
