Les dix vecteurs · quatre familles

FAMILLE A Tool layer descriptions empoisonnées
  • 01TPA — description statique · instruction cachée dans le JSON.
  • 02Rug pull · description propre à l'install, malveillante après push.
  • 03Unicode tags + line-jumping · caractères invisibles ou faux séparateurs.
FAMILLE B Prompt layer injection cross-document
  • 04Injection cross-document · issue, mail, doc → tool call non sollicité.
  • 05Confused deputy · l'agent agit avec ses droits sur instruction non-fiable.
  • 06Exfil via metadata · commit message, PR titre, log accessible.
FAMILLE C Cross-server collision & shadowing
  • 07Namespace collision · deux serveurs exposent un outil de même nom.
  • 08Shadowing + capability leak · l'outil A se substitue à l'outil B.
FAMILLE D Transport · identité OAuth, supply chain
  • 09OAuth / DCR pitfalls · open registration, token passthrough, session hijack.
  • 10Supply chain NPM/PyPI · typosquatting, compromission upstream, dépendance transitive.

La matrice défensive · dix patterns

Load-bearing · 01
Signature Sigstore + hash pinning
Couvre A · DTPA statique, rug pull, Unicode tags, compromission upstream, dépendance transitive — la confiance se déplace du runtime vers l'install.
Load-bearing · 02
Tool tagging au runtime (read · write · external)
Couvre B · CLimite la blast radius des injections cross-document et la confusion entre serveurs.
Load-bearing · 03
Allowlist namespace par serveur
Couvre CPréfixage forcé + déclaration explicite — élimine la famille C.
Load-bearing · 04
Human-in-the-loop sur les write tools
Couvre BAucune écriture externe sans confirmation humaine — le prix UX d'éliminer la famille B.

Le pivot — automne 2026
La spec MCP v2 inverse la logique : signature à l'install, plus audit au runtime.
L'industrie a passé 18 mois à patcher MCP en runtime — sandboxes, garde-fous, humans in the loop. La révision v2, en discussion à l'AAIF, intègre Sigstore obligatoire sur les registries officiels, tool tagging structurel, allowlist namespace et schéma d'événements OTel. Les déploiements qui ne migrent pas resteront sur du runtime patching à perpétuité.

Règle de pouce · 2026

Quatre phrases pour partir en production sans pivoter dans un an.

  1. Signer ou pinner — aucun serveur MCP sans Sigstore ou hash pinné dans mcp-config.json.
  2. Tagger les outils — read_only, external_write, local_write dès l'install.
  3. Consent humain sur toute écriture hors-périmètre — la friction est le prix de la famille B.
  4. OTel dès le jour 1 — sans audit structuré, l'incident response est de la divination.
